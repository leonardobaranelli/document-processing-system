# AI Usage Policy

This document fulfills the **"AI Usage Policy"** section of the take-home challenge. It discloses,
in full transparency, every way in which AI tools were used while building this project, and keeps
a running history of prompts and generated code.

---

## 1. Tools declared

| Tool | Purpose | Access mode |
| --- | --- | --- |
| Cursor IDE + Anthropic Claude (coding agent) | Pair-programming assistant used to scaffold modules, write comprehensive boilerplate, draft documentation, and produce sample text files. | Local IDE with remote LLM inference (no project secrets shared). |

## 2. Which code was generated / assisted by AI

The project was produced in collaboration with a Claude-powered coding agent. All of the code,
configuration, and documentation in this repository was either drafted or proposed by the agent
and then reviewed and edited by the human author before being committed. Specifically:

- **Backend (NestJS)** — module scaffolding, Prisma schema, BullMQ workers, controllers, DTOs,
  WebSocket gateway, global filters/interceptors, and configuration. The core AI engine
  (`mlp.ts`, `text-rank.ts`, `feature-extractor.ts`, `ai.service.ts`) was written with AI
  assistance; all numerical logic (forward pass, backpropagation, similarity function,
  TextRank power iteration) was reviewed line by line by the author.
- **Frontend (React + Vite + Tailwind)** — component scaffolding, API client, Socket.IO
  integration, and the dashboard / process-detail pages.
- **Infrastructure** — `docker-compose.yml`, service `Dockerfile`s, `nginx.conf`,
  `.env.example`, `.gitignore`.
- **Documentation** — this `AI_USAGE_POLICY.md`, `ARCHITECTURE.md`, `API_DOCS.md`, and the
  root `README.md`.
- **Sample data** (`sample-data/01-*.txt` … `12-*.txt`) — the 500+ word texts on multiple
  topics were generated by the assistant to serve as realistic input for the processing
  pipeline. They are original paraphrases of well-established public knowledge; no copyrighted
  passages were reproduced.

Everything else (final review, integration choices, commit discipline, design decisions,
secrets, and environment configuration) was controlled by the human author.

## 3. Justification of AI usage per area

| Area | Why AI was used | Author's role |
| --- | --- | --- |
| Boilerplate / scaffolding | Removes repetitive typing so the author can focus on design and correctness. | Approves module boundaries, naming, and patterns. |
| Prisma schema | Quick exploration of relational choices for processes, documents, analyses, and logs. | Validates normalization, indexes, and cascade rules. |
| BullMQ worker | AI suggested the concurrency + cooperative-cancellation skeleton. | Adjusts state machine checks to avoid race conditions. |
| MLP / TextRank code | AI was used to accelerate writing well-known algorithms; both are public-domain techniques. | Reviews math, activation derivatives, init strategy, and test coverage. |
| Sample text files | Quickly produce 10+ realistic documents without copyright risk. | Verifies word count and topical diversity. |
| Documentation | Produce first drafts that the author then edits. | Ensures accuracy and removes AI filler. |

## 4. Modifications made to AI-generated code

The following kinds of modifications were applied to the AI output as a matter of course:

- **Correctness fixes**: tightening types, adding optimistic-locking `where` clauses on Prisma
  `updateMany`, correcting backpropagation indices, ensuring `normalize` handles degenerate
  ranges.
- **Style/lint conformance**: aligning with the Prettier + ESLint configuration (single
  quotes, trailing commas, sorted imports).
- **Security hardening**: adding Helmet, strict `ValidationPipe` with `forbidNonWhitelisted`,
  parameterized queries only (Prisma), CORS allow-list via env var.
- **Documentation tone**: removing promotional language, trimming redundant sections, and
  clarifying the distinction between mandatory challenge requirements and bonus items.
- **Commit discipline**: all commit messages are written by the human author in lowercase
  `type(scope): subject` Conventional Commits style, independent of whatever the AI proposed.

El scoring del MLP se alimenta de un extractor de features por oración (posición, longitud, densidad léxica, overlap con top words, TF-IDF y demás). Dashboard con listado de procesos en vivo, formulario para arrancar un nuevo analisis, y página de detalle con barra de progreso, estadísticas, controles pause/resume/stop, nube de top words, resumen global y activity log en streaming via WebSocket. React Query para el caché de estado y socket.io client para los updates en tiempo real.
>
> Infra: Dockerfiles multi-stage (development, build, production) para backend y frontend, y un docker-compose.yml que levante postgres, redis, backend y frontend con healthchecks. El frontend en prod detrás de nginx. .env.example documentado en la raíz.

> Datos de prueba: genera unos 12 archivos .txt de 500+ palabras cada uno, para demostrar el sistema, sobre temas variados (preferiblemente de tecnología), montados read-only en el contenedor del backend.

### Session 2 — Per-document summaries + global summary-of-summaries

>
> Ahora debes encargarte de lo siguiente:
> El análisis de los documentos actualemente está haciendo un resumen global de la convergencia de todos los archivos que se procesan en conjunto, en su lugar el módulo de ia debe encargarse de hacer un resumen individual de cada archivo (cada archivo por mas que se procese en conjunto con otros debe tener su tratamiento individual, y en consucuencia su data relativa a eso almacenada también), y el resumen global debe ser un resumen de todos los resumenes individuales.

**AI-generated output in this session:**
- `backend/src/modules/ai/ai.service.ts` — refactored `aggregate()` so the global summary re-runs TextRank + MLP over the union of per-document summary sentences; extracted `summarizeCorpus()` helper shared by single-document and cross-document scoring.
- `backend/src/modules/ai/ai.service.spec.ts` — added tests that prove the global summary is an extractive selection from the per-document summary pool and that it degrades gracefully when there are no summary sentences.
- `backend/src/modules/process/dto/process-response.dto.ts` — introduced `PerDocumentAnalysisDto` and `ProcessResultsDetailDto` (extends `ProcessResultsDto`).
- `backend/src/modules/process/process.controller.ts` — `GET /process/results/:id` now returns `ProcessResultsDetailDto`.
- `backend/src/modules/process/process.service.ts` — `getResults()` joins `DocumentAnalysis` and maps each document into `per_document[]`.
- `frontend/src/lib/api.ts` — new `PerDocumentAnalysis` / `ProcessResultsDetail` types; `getResults()` retyped.
- `frontend/src/pages/ProcessDetailPage.tsx` — dedicated query for `/results/:id` and a new "Per-document analyses" section with stats, top words and per-file summaries.
- `docs/API_DOCS.md`, `docs/ARCHITECTURE.md` — documented the `ProcessResultsDetail` shape and the summary-of-summaries design.
